2. **`apply_patch.js` の改良**:

3. **ハッシュチェック（CRC検証）の設計 (00:23)**:
   - ユーザー様より「バックアップ前にオリジナルハッシュを検証し不一致なら中止」「適用後もハッシュ検証し不一致ならバックアップから自動復帰」との高度なセーフティネット追加の指示を受領し、承認。
   - 最新純正の `app.asar` のSHA-256ハッシュを `Get-FileHash` で測定し、基準値 `7EBE22606F03EAA7CDD0A0384B6240A0E5E20064C5EFCC0DA446A38F33AAC74B` を特定。
4. **PowerShellパースエラーとの対峙**:
   - `install.ps1` 実行時に、`At line:29 char:1 Unexpected token '}'` や `The string is missing the terminator: "` などの難解な構文エラーが多発。
   - 原因調査1：BOMなしのUTF-8をWindows PowerShell 5.1がShift_JISとして読み込んだため、日本語全角文字のバイト列が制御文字と誤認された。
   - 対処1：PowerShellを使い「BOM付き UTF-8」に強制変換するスキームを導入。
   - 原因調査2：それでもエラーが継続。詳細にスキャンした結果、`cmd /c` の引数内でエスケープしたダブルクォーテーションの閉じ漏れ（`"`"$NewAsar`"` の末尾）を発見。これがスクリプトの残りをすべて巨大文字列として飲み込んでいた。
   - 対処2：エスケープバグを100%排除するため、`cmd /c` 形式を廃止し、PowerShell標準の `Start-Process` と引数配列渡し（シングルクォーテーションによる安全な文字列結合）に変更。
   - 原因調査3：まだ同様のパーサーエラーが別行で発生。最終的に、文字コード誤認（Shift_JISパース）による全角文字の干渉を根本から排除するため、コンソールの表示ログをすべて「オールASCII・英語メッセージ」に変更。
   - **結果**: この「エスケープフリー ＋ オールASCII」の究極の安全設計により、PowerShellのパースエラーを完璧に克服。
5. **最終テストの成功 (15:33)**:
